## Latar Belakang Cerita

**PT. Tirta Jaya** adalah distributor minuman yang memiliki **2 gudang**:

1. **Gudang Cikarang** - Gudang utama, lebih besar
2. **Gudang Bandung** - Gudang cabang, lebih kecil

**Haikal** adalah developer sistem WMS. Dia yang membuat user baru untuk setiap gudang dan melakukan maintenance sistem.

Setiap gudang punya:
- 1 **Admin Cabang** â†’ Kepala gudang, kelola master data, rekonsel stock opname, lihat history
- 1-2 **Admin Warehouse** â†’ Staff operasional, inbound, outbound, input stock opname

### Batasan Fitur Edit & Batal

| Batasan | Keterangan |
|---------|------------|
| **Window time** | Hanya transaksi hari ini (tanggal = CURRENT_DATE) |
| **Role** | Hanya admin_warehouse yang bisa edit/batal |
| **Scope** | Per warehouse (tidak bisa edit transaksi gudang lain) |
| **Audit trail** | Saat ini hard delete (tidak ada log pembatalan) |
| **Alasan** | Tidak diminta (opsional: bisa ditambah field reason) |

### Rekomendasi Database untuk Production

Jika ingin audit trail yang lebih baik, tambahkan field berikut:

```sql
-- Untuk soft delete dan audit trail
ALTER TABLE inbound_history ADD COLUMN is_cancelled BOOLEAN DEFAULT FALSE;
ALTER TABLE inbound_history ADD COLUMN cancelled_at TIMESTAMP;
ALTER TABLE inbound_history ADD COLUMN cancelled_by UUID REFERENCES users(id);
ALTER TABLE inbound_history ADD COLUMN cancellation_reason TEXT;

-- Sama untuk outbound_history
ALTER TABLE outbound_history ADD COLUMN is_cancelled BOOLEAN DEFAULT FALSE;
ALTER TABLE outbound_history ADD COLUMN cancelled_at TIMESTAMP;
ALTER TABLE outbound_history ADD COLUMN cancelled_by UUID REFERENCES users(id);
```

| No | ID Transaksi | Tanggal | Pengemudi | No. Polisi | Kode Produk | Nama Produk | Lokasi # | Lokasi | BB Produk | Expired Date | Status | Waktu Input |

**Note Export Excel:**
- Format: 1 baris per lokasi (bukan per transaksi)
- Satu ID Transaksi bisa memiliki banyak baris (sesuai jumlah lokasi FEFO)
- Kolom "Lokasi #" menunjukkan urutan lokasi dalam transaksi
- BB Produk dan Expired Date per lokasi untuk verifikasi FEFO

### SCHEMA vs KONDISI SAAT INI (Outbound History)

| Aspek | Schema (outbound_history) | Kondisi Frontend Saat Ini |
|-------|---------------------------|---------------------------|
| **BB Produk Storage** | `bb_produk varchar(20)` - 1 per record | Multiple BB Produk dari `locations` FEFO |
| **Locations** | `locations jsonb` dengan `stock_id`, `expired_date` | Array string lokasi, BB diambil dari `stock_list` |
| **Export Format** | 1 baris per transaksi | 1 baris per lokasi (lebih detail) |
| **QR Code** | Tidak ada di schema | Frontend generate QR dari BB Produk |

**Perubahan yang Diperlukan untuk Database:**
1. Schema `outbound_history.locations` perlu menyimpan `bb_produk` per lokasi
2. Atau, join dengan `stock_list` saat query untuk mendapatkan BB Produk

```sql
-- Option 1: Update locations jsonb structure
locations: [
  {
    "cluster": "A",
    "lorong": 1,
    "baris": 3,
    "level": 1,
    "qty_carton": 48,
    "stock_id": "uuid",
    "bb_produk": "2512210001",  -- Tambahkan field ini
    "expired_date": "2025-12-21"
  }
]

-- Option 2: JOIN dengan stock_list saat query
SELECT o.*, s.bb_produk, s.expired_date
FROM outbound_history o
LEFT JOIN stock_list s ON s.id = ANY(o.locations->>'stock_id')
```
